# 自动修改逻辑

自动修改的目标是减少机械返工，不是替供应商创造事实或测试。

---

## 一、允许自动修改

### 确定性名称错误

- 错误品牌名；
- 错误产品名；
- 已批准的固定替换。

### 基础语言问题

- 错别字；
- 标点；
- 空格；
- 重复词；
- 明显语病；
- 格式；
- 标签完全重复。

### 安全语义收敛

仅在规则明确批准时：

- “所有交通方式任意组合”收敛为“支持组合公交、骑行、打车等多种出行方式”；
- 无依据“最优路线”可生成“推荐方案”的建议稿。

安全语义收敛原则：

1. 不新增事实；
2. 不改变数字；
3. 不改变测试结果；
4. 不改变产品对象；
5. 不将待确认功能替换成另一项已确认功能；
6. 不隐瞒原问题。

---

## 二、禁止自动修改

- 数字；
- 日期；
- 时间；
- 城市；
- 金额；
- 比例；
- 测试结果；
- 测试数量背后的真实场景；
- 功能是否存在；
- 功能上线范围；
- 页面操作流程；
- 对比结论；
- 证据内容；
- 隐私和授权问题；
- 项目资料冲突；
- 作者核心判断；
- 任何需要供应商补充的信息。

---

## 三、修改流程

```text
识别问题
→ 检查规则是否允许自动修改
→ 检查字段是否属于禁止字段
→ 生成局部建议
→ 验证未新增事实
→ 生成原文/修改稿 Diff
→ 等待供应商或内部人员确认
→ 保存新内容版本
→ 重新审核
```

---

## 四、修改验证

修改 Agent 输出后，需要再次运行：

1. 数字一致性检查；
2. 专有名词检查；
3. 新增事实检查；
4. 未确认功能检查；
5. 标题正文一致性；
6. 测试数量一致性；
7. 必带规则检查。

修改稿不能因为是 AI 生成就跳过审核。

---

## 五、局部修改优先

默认只修改问题所在句子或段落。

除非内容结构严重混乱，不应整篇重写。

整篇重写需要：

- 显示完整 Diff；
- 明确哪些事实保持不变；
- 标出需要供应商补充的位置；
- 人工确认后才能保存。

---

## 六、版本记录

每次修改保存：

- 原内容版本；
- 修改后版本；
- 修改 Agent；
- Prompt 版本；
- 模型；
- 修改规则编号；
- 修改原因；
- 确认人；
- 确认时间；
- 是否被再次修改。
